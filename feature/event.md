# Event Loop Design (Linux)

## Goals
- Linux-only implementation based on `epoll` + `eventfd`.
- Multi-producer, single-consumer notify queue (MPSC) for cross-thread scheduling.
- Loop-local defer queue for loop-thread scheduling and cancellation.
- Strictly asynchronous execution on the loop thread (no inline execution during `post`).
- Timers integrated into a single event loop.
- Fixed callbacks via handle-member entries (no per-post allocations).

## Threading Model
- One event loop thread owns the poller, timer heap, and local defer queue.
- Any thread may call `post(...)` for notify entries and `stop()`.
- Only the loop thread may call `post_local(...)`, `cancel(...)` for local entries,
  and `post_at(...)`/`cancel(...)` for timers (asserted).
- Cross-thread calls enqueue into the notify MPSC and signal `eventfd` to wake the loop.

## Loop Group
`EventLoopGroup` owns a fixed set of loops and a `ThreadGroup`. `start()` runs one
loop per thread and installs the loop's coroutine frame pool for that thread.
Callers choose a loop via `at(index)`.

## Wakeup Strategy
- `eventfd(EFD_NONBLOCK | EFD_CLOEXEC)` is registered in `epoll`.
- Producers write `uint64_t(1)` to `eventfd` to wake the loop.
- The loop drains `eventfd` before processing queued commands.
- An atomic flag (`wakeup_pending_`) prevents redundant wakeups.

## Notify Queue (MPSC)
`NotifyEntry` provides an intrusive, fixed-callback scheduling primitive:
- `post<Handle, EntryMember, RunCb>(handle)` enqueues the entry for execution on
  the loop thread.
- Entries have a single callback (`on_run`) generated by a template trampoline.
- There is no built-in cancellation; callbacks should check handle state if
  cancellation is required.
- Posting from the loop thread does not wake the loop.

## Local Defer Queue (Loop-only)
`DeferEntry` is a loop-local scheduling primitive:
- `post_local<Handle, EntryMember, RunCb>(handle)` enqueues on the loop thread.
- If the entry is already queued, the post is ignored (idempotent enqueue).
- `cancel<Handle, EntryMember>(handle)` removes the entry if still queued.
- Used for callbacks that touch awaiter memory or coroutine handles.

## Awaiter Timeout + Cancellation Safety (Design Rules)
- Awaiter memory is only accessed on the loop thread.
- Timeout is implemented as a wrapper awaiter (`TimeoutAwaiter`) around a
  concrete awaiter (no helper coroutine).
- `await_suspend` arms a timer only if the wrapped awaiter suspends.
- Timeout callback resumes the coroutine via loop-local `DeferEntry` to avoid
  nested resume inside timer processing.
- `await_resume` cancels the timer on normal completion and returns either the
  wrapped result or a timeout error.
- `TimeoutAwaiter` destructor cancels its timer; the wrapped awaiter destructor
  must cancel its own side effects so no further resume is possible.
- Cross-thread notifications use `NotifyEntry` and must not capture awaiter
  pointers; callbacks must observe canceled state and drop work if needed.

## TimerQueue (Heap)
`TimerQueue` is a C++ translation of `libuv`'s `heap-inl.h`, used as an intrusive
min-heap for timer nodes. It provides heap primitives and does not own timer data.

```cpp
class TimerQueue {
public:
    struct Node {
        Node *left;
        Node *right;
        Node *parent;
    };

    void init();
    Node *min() const;
    std::size_t size() const;
    bool empty() const;

    void insert(Node *node);
    void remove(Node *node);
    void dequeue();
};
```

## API Summary
```cpp
namespace fiber::event {

class EventLoop {
public:
    void run();
    void run_once();
    void stop();

    template<typename Handle, auto EntryMember, auto RunCb>
    void post(Handle &handle) noexcept;

    template<typename Handle, auto EntryMember, auto RunCb>
    void post_local(Handle &handle) noexcept;

    template<typename Handle, auto EntryMember>
    void cancel(Handle &handle); // DeferEntry or TimerEntry

    template<typename Handle, auto EntryMember, auto Cb>
    void post_at(std::chrono::steady_clock::time_point when, Handle &handle);
};

class EventLoopGroup {
public:
    explicit EventLoopGroup(std::size_t size);

    void start();
    void start(const fiber::async::SignalSet &mask);
    void stop();
    void join();

    EventLoop &at(std::size_t index);

};

} // namespace fiber::event
```

## Loop Cycle
1) Update `now_`, run due timers.
2) Drain notify queue (all pending entries).
3) Drain local defer queue (all pending entries).
4) `epoll_wait` with timeout from the next deadline.
5) Dispatch IO callbacks.
6) Drain local defer queue once (callbacks scheduled by IO).

## Implementation Notes
- All loop-side execution is strictly async (no inline execution in `post`).
- The poller, timer heap, and local defer queue are loop-thread only.
